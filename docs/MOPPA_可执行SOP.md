# MOPPA 可执行 SOP（v1.0）

## 0. 目标与范围
- 目标：建立可重复、可审计、可量化的数据管道评测体系，覆盖“事件获取 -> 出题 -> 预测 -> 真值回填 -> 评分 -> 排行与反馈”。
- 范围：L1/L2/L3/L4 全等级问题、自动与人工协同流程、模型评测与榜单输出。
- 不在范围：模型训练细节、外部真值系统内部实现。

## 1. 角色与职责（RACI）
- 情报专家（A/R）：确认模板、判定问题质量规则、抽检评分质量。
- 算法工程师（R）：事件过滤、自动出题、评分策略实现与优化。
- 平台工程师（R）：调度、存储、幂等、告警、权限与审计。
- 运营/审核（R）：人工筛题、评论治理、社区反馈闭环。
- 数据分析（C）：指标看板、周/月报、偏差分析。

## 2. 数据契约（必须版本化）

### 2.1 通用字段（所有实体必须包含）
- `id`：全局唯一 ID（UUID 或雪花 ID）。
- `version`：schema 版本号（如 `event.v1`、`question.v2`）。
- `source`：来源系统标识。
- `created_at` / `updated_at`：UTC 时间戳。
- `trace_id`：链路追踪 ID（贯穿全流程）。

### 2.2 核心实体
- `event`：事件原文、时间、来源可信度、分类标签。
- `question`：题干、等级（L1-L4）、截止时间、模板 ID、可验证条件。
- `prediction`：模型 ID、预测内容、置信度、推理耗时、提交时间。
- `ground_truth`：真实答案、证据链接、发布时间、可信度等级。
- `score`：维度得分、总分、权重版本、评分方式（rule/ai/hybrid）。

## 3. 端到端流程与阶段门（Stage Gate）

## 3.1 阶段 S0：初始化与配置
- 输入：系统部署完成、管理员账号、数据源接入信息。
- 动作：
  - 配置数据源连接（只读账号、最小权限）。
  - 配置规则等级（L1-L4）与问题模板。
  - 配置参与评测模型（URL、Key 引用、限流策略）。
  - 配置评分规则版本。
- 准出标准（DoD）：
  - 配置项通过连通性检查；
  - 密钥未明文落库/落文档；
  - 产出配置快照（含版本号与变更人）。

## 3.2 阶段 S1：事件采集与过滤
- 输入：数据源事件流。
- 动作：
  - 拉取事件并标准化；
  - 按规则过滤无效/低价值事件；
  - 记录过滤原因标签。
- 准出标准：
  - 有效事件入库率 >= 95%；
  - 过滤原因可追溯（每条事件至少 1 个原因码）。

## 3.3 阶段 S2：自动出题与答案空间生成（L1/L2 可选）
- 输入：有效事件 + 模板。
- 动作：
  - 基于模板自动生成问题；
  - L1/L2 可生成候选答案空间（自动/人工）。
- 准出标准：
  - 题目包含截止时间、可验证条件；
  - 重复率 <= 5%（7 天滚动窗口）；
  - 不可判定题占比 <= 3%。

## 3.4 阶段 S3：人工质检与发布
- 输入：候选问题池。
- 动作：
  - 审核队列处理（通过/驳回/修改）；
  - 驳回必须填写原因标签（模板问题/时效性/不可验证/重复等）。
- 准出标准：
  - 抽检一致性 >= 90%（双人复核样本）；
  - 发布问题均具备可验证证据路径。

## 3.5 阶段 S4：模型预测执行
- 输入：已发布问题。
- 动作：
  - 按调度策略调用各模型预测；
  - 结果按统一格式写入 `prediction`。
- 准出标准：
  - 每日任务按 SLA 完成（例如 30 分钟内）；
  - 模型执行成功率 >= 98%；
  - 失败任务有可重试记录。

## 3.6 阶段 S5：真值回填
- 输入：到期问题列表。
- 动作：
  - 按答案发布时间拉取真实答案；
  - 写入证据链接并标注可信度。
- 准出标准：
  - 真值覆盖率 >= 97%；
  - 延迟回填率（超 24h）<= 2%。

## 3.7 阶段 S6：评分与榜单
- 输入：`prediction` + `ground_truth` + 评分规则。
- 动作：
  - 按题型与等级打分；
  - 输出分榜（L1/L2/L3/L4）与总榜（周/月）。
- 准出标准：
  - 评分可复现（同输入同结果）；
  - 权重版本可追溯；
  - 生成周报/月报。

## 3.8 阶段 S7：社区反馈闭环
- 输入：榜单与问题页。
- 动作：
  - 开放评论与人工预测；
  - 治理违规内容；
  - 沉淀改进项（模板、规则、评分）。
- 准出标准：
  - 反馈问题 100% 有处理状态；
  - 每周输出 Top 5 可执行改进项。

## 4. 调度、幂等与重试策略
- 调度：每日主任务 + 每小时补偿任务。
- 幂等键：`task_type + business_id + date_window + model_id`。
- 重试策略：指数退避（1m/5m/15m），最多 3 次；超过阈值进入死信队列。
- 超时控制：单模型超时（如 120s），单批次超时（如 30m）。
- 补偿策略：仅处理“失败且未完成”的任务，禁止重复写入。

## 5. 指标体系（最小可用集）
- 流程效率：端到端时延、各阶段处理时长、每日完成率。
- 数据质量：重复率、不可判定率、真值覆盖率、回填延迟率。
- 模型效果：按等级准确率、Brier Score（如适用）、稳定性波动。
- 运营质量：人工审核一致性、驳回原因分布、社区反馈处理时效。

## 6. 安全与合规（强制）
- 密钥管理：全部使用环境变量或密钥服务（Vault/KMS），禁止明文。
- 访问控制：RBAC（管理员/审核员/只读分析）最小权限。
- 审计日志：配置变更、评分规则变更、手工改题必须留痕。
- 数据脱敏：日志中禁止输出敏感字段（账号、口令、完整 token）。

## 7. 异常处理 Runbook（简版）
- P0（全链路中断）：15 分钟内拉群，30 分钟给出临时恢复方案。
- P1（核心阶段失败率 > 10%）：1 小时内修复或降级。
- P2（单模型异常）：隔离模型，主流程继续，补偿任务回补。
- 任何异常必须记录：发生时间、影响范围、根因、修复动作、预防措施。

## 8. 变更管理
- 规则/模板/权重变更必须走评审，记录 `change_id`。
- 每次变更需灰度验证（例如 10% 样本、1 天观察窗口）。
- 失败回滚：可在 10 分钟内回滚至上一稳定版本。

## 9. 交付物清单（每周）
- 周榜与分榜（L1-L4）。
- 指标看板快照（效率、质量、效果、运营）。
- 异常与改进报告（含负责人和截止时间）。

## 10. 上线前检查清单（Go-Live Checklist）
- [ ] 所有密钥已脱敏，文档无明文凭据。
- [ ] 各阶段 DoD 已配置并可自动校验。
- [ ] 幂等键、重试、死信队列已启用。
- [ ] 告警规则已联通（失败率、延迟、覆盖率）。
- [ ] 周报模板和责任人已确定。
